# Alpha 测试通过线

> 版本 v1.0 | 2026-05-24

---

## 一、通过线定义

Alpha 测试的"通过"不意味着产品完美，而是指产品达到了**可进入小范围真实用户测试的底线**。

---

## 二、必要条件（AND 关系，全部通过才算通过）

### A. 安装门槛

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| 安装成功率 | ≥ 80% | 成功安装人数 / 尝试安装人数 |
| SmartScreen 放弃率 | ≤ 20% | 因安全提示放弃的人数 / 遇到提示的人数 |
| 安装平均耗时 | ≤ 10 分钟 | 从下载到成功启动 |

**不通过判定**：如果 ≥ 40% 的用户卡在安装步骤且无法自行解决，Alpha 不通过。

### B. 首次体验

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| 3 分钟内看到 K 线 | ≥ 70% | 加载演示数据后自动展示 K 线 |
| 独立完成至少 4/6 任务 | ≥ 60% | 测试脚本任务 1-6 |
| 报告"不知道干嘛" | ≤ 30% | 反馈问卷 |

### C. 核心功能可用性

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| K 线加载无明显卡顿 | ≥ 80% | 反馈问卷 |
| 数据导入成功率 | ≥ 80% | 成功导入人数 / 尝试导入人数 |
| 回测执行成功率 | ≥ 90% | 运行成功次数 / 运行总次数 |
| 回测结果合理性 | ≥ 70% 用户认可 | 反馈问卷"有没有不合理的交易记录" |

### D. 合规感知

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| 理解"不荐股" | ≥ 80% | 反馈问卷 Q19 |
| 看到风险提示 | ≥ 70% | 反馈问卷 Q20 |
| 不会认为"这是荐股软件" | ≤ 10% | 反馈问卷 + 访谈 |

### E. 信任度

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| 担心病毒比例 | ≤ 30% | 反馈问卷 Q21 |
| 担心数据上传比例 | ≤ 20% | 反馈问卷 Q22 |
| 推荐意愿均分 | ≥ 6/10 | 反馈问卷 Q23 |

### F. 付费意愿

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| 付费意愿均分 | ≥ 5/10 | 反馈问卷（1-10 分） |
| 愿意付 ≥ ¥150 比例 | ≥ 40% | 反馈问卷 Q18 |

### G. 技术稳定性

| 指标 | 通过线 | 测量方式 |
|------|:---:|------|
| CRITICAL Bug 数 | = 0 | Bug 追踪表 |
| HIGH Bug 数 | ≤ 5 | Bug 追踪表 |
| 崩溃次数 / 用户 | ≤ 1 | 用户反馈 + 日志 |

## 三、通过判定规则

### Alpha 通过

```
A 安装门槛（全部） + B 首次体验（全部） + 
C 核心功能（全部） + D 合规感知（全部） + 
E 信任度（全部） + F 付费意愿（全部） + 
G 技术稳定性（全部）
= 全部通过 → Alpha 通过 → 可进入小范围销售
```

### Alpha 有条件通过

```
大部分通过线达到，但 1-2 个指标差 ≤ 5% 未达标 →
修复后重新测试未达标的指标 →
通过后进入小范围销售
```

### Alpha 不通过

```
≥ 3 个指标未达标，或任一 CRITICAL 条件（如安装成功率 < 50%）未达标 →
分析原因 → 修复 → 重新招募测试 →
Alpha 测试第二轮
```

## 四、数据收集与判定

### 最低样本量

| 条件 | 要求 |
|------|:---:|
| 最低测试用户数 | 8 |
| 完成完整反馈问卷 | ≥ 6 |
| 如不足 8 人 | 延长招募，不降标准 |

### 反馈质量要求

- 每个问题至少 50% 用户给出了有效回答（非空/非"不知道"）
- 必须包含至少 1 条具体的负面反馈或改进建议
- 如果所有测试用户都说"完美无问题"，视为无效测试（需替换测试用户）
