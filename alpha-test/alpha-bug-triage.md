# Alpha Bug 分级标准

> 版本 v1.0 | 2026-05-24

---

## 一、严重性分级

### CRITICAL — 阻塞发布

**定义**：导致数据丢失、软件崩溃、安全漏洞、或合规风险的问题。

| 示例 |
|------|
| 回测结果计算错误，导致严重误导 |
| 数据导入导致数据库损坏或数据丢失 |
| 软件启动即崩溃（启动成功率 < 80%） |
| 激活码验证被绕过（安全漏洞） |
| 内存泄漏导致 30 分钟内软件不可用 |
| 用户数据被意外删除（任何路径） |
| 合规文案出现荐股/收益承诺表达 |

**处理**：必须修复后才能发布，不允许带 CRITICAL 发布。

---

### HIGH — 重要用户体验问题

**定义**：严重影响核心功能使用体验，但不会导致数据丢失或崩溃。

| 示例 |
|------|
| K 线图在特定操作下加载失败 |
| 通达信数据导入成功率 < 80% |
| 回测执行效率极低（> 30 秒/次） |
| 重要页面无空状态提示，为空白 |
| 错误提示为英文/技术术语（用户看不懂） |
| 某核心功能入口找不到 |
| 演示数据加载失败 |

**处理**：应修复后再发布。如确实无法修复，必须在 `known-issues.md` 中诚实列出。

---

### LOW — 非阻塞性问题

**定义**：影响体验但不影响核心功能使用的问题。

| 示例 |
|------|
| UI 文案有错别字 |
| 某个配色/间距不一致 |
| 非核心功能缺少空状态 |
| 某个按钮点击无反馈 |
| 性能可优化但当前不影响使用 |
| 某个非关键操作无加载状态 |

**处理**：记录到 issue tracker，可延后修复。

---

## 二、影响范围分级

| 级别 | 影响用户比例 |
|:---:|------|
| 全量 | 100% 用户 |
| 大部分 | 60-99% 用户 |
| 部分 | 20-59% 用户 |
| 少数 | 5-19% 用户 |
| 极少数 | < 5% 用户 |

严重性 × 影响范围 = 实际优先级。

## 五、修复后验证

每个 CRITICAL/HIGH Bug 修复后：

1. [ ] 原场景复测通过
2. [ ] 相关功能回归测试通过
3. [ ] 无引入新 Bug
4. [ ] 更新 `known-issues.md`（如适用）

---

> **原则**：对 Bug 诚实。不捂盖子、不大事化小。用户比你更敏感，隐瞒只会破坏信任。
