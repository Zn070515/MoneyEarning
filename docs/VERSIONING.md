# 版本号规则

遵循 **Semantic Versioning 2.0.0** 精简版，格式 `MAJOR.MINOR.PATCH`。

## 变更判定

| 变更类型 | 版本位 | 示例 |
|---------|--------|------|
| Bug 修复、UI 微调、性能优化、重构、文档 | **PATCH** (第三位) | 0.10.0 → 0.10.1 |
| 新功能、新页面、新模块、新指标 | **MINOR** (第二位) | 0.10.1 → 0.11.0 |
| 破坏性变更（API 不兼容、数据格式不兼容、架构重写） | **MAJOR** (第一位) | 0.11.0 → 1.0.0 |

## 黄金规则

1. **小改动坚决只动第三位。** 不要为了"看起来像大更新"而跳第二位。
2. **新功能才动第二位，同时第三位归零。**
3. **MAJOR 在 1.0.0 之前保持 0。** 进入正式发行后，破坏性变更才升第一位。
4. **一次 commit 只改一个版本位。** 如果既有 bug 修复又有新功能，新功能优先 → 动第二位。

## 需要同步的文件（三处保持一致）

- `packages/app/package.json` — `"version"`
- `packages/app/src-tauri/tauri.conf.json` — `"version"`
- `packages/app/src-tauri/Cargo.toml` — `version`

## 示例场景

```
0.10.0 → 0.10.1   修复 44 个 bug，无新功能
0.10.1 → 0.11.0   新增某策略模板 + 优化扫描引擎
0.11.0 → 1.0.0    产品正式发布（非破坏性，但里程碑意义）
```
