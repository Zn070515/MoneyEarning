# 回测正确性审计

> 版本 v1.0 | 2026-05-24 | 审查范围：`crates/wasm-backtest/` + `packages/ui/src/BacktestPanel.tsx`

---

## 一、未来函数风险检查

### 检查 1：信号与成交时间对齐

**风险**：如果在 bar[i] 上计算的信号使用了 bar[i] 的收盘价，然后以 bar[i] 的收盘价成交 → 未来信息泄露（在真实交易中，你无法在收盘前知道收盘价）。

**当前实现**（`strategies.rs`）：
- 所有信号计算在 `generate_signals(data)` 中完成，使用完整的 OHLCV 数据
- `simulate()` 在逐 bar 循环中按顺序访问信号和价格 — 信号在 bar[i] 生成时就访问了 close[i]
- **结论**：❌ **存在未来函数风险**。信号使用了当日收盘价，然后假设以当日收盘价成交。

**建议修复**：
- 方案 A：所有信号改为使用 `close[i-1]`（前一日收盘价确认）+ 次日开盘价成交
- 方案 B：明确在回测报告中标注"假设以当前 bar 收盘价成交"，并注明此为回测乐观偏差来源

**当前状态**：方案 B 更务实 — 在回测报告中明确声明此假设。

### 检查 2：技术指标信号滞后

**风险**：`sma_vec`, `ema_vec`, `macd_vec` 等辅助函数是否在信号中使用了未来信息？

**当前实现**：
- 这些函数对整个数据序列进行计算后返回 `Vec<f64>`
- 在 `generate_signals()` 中，所有值已预先计算完毕
- 当 `simulate()` 逐 bar 查询 `signals[i]` 时，信号值是基于 bar[i] 时刻可见数据计算的
- **结论**：✅ 无未来函数（正确的逐 bar 滚动窗口计算），除非函数本身有实现错误

### 检查 3：前向偏移引用

**风险**：代码中是否出现 `data[i+1]` 或 `shift(-1)` 等未来引用？

**当前实现**：
- `simulate()` 循环：`for i in 0..n`，仅访问 `signals[i]`, `prices[i]`, `equity[i]`
- 策略信号计算中仅使用 `close[i]`, `high[i]`, `low[i]`, `volume[i]`
- `ref(data, 1)` 正确使用 `data[i-1]`（过去值）
- **结论**：✅ 无前向引用

## 二、成交价格可实现性

### 检查 4：涨停买入

**风险**：在涨停日以涨停价成交的买入在实际中几乎无法执行。

**当前实现**：
- `simulate()` 不检查涨跌停状态
- 策略信号在涨停日（`close == high && high > ref(close,1) * 1.10`）仍然生成 `Buy` 并成功成交
- **结论**：❌ **未处理涨跌停**。回测结果可能偏乐观。

**建议修复**：
- 在 `simulate()` 中加入涨停/跌停检测
- 涨停日：买入信号 → 延迟到次日（或直接跳过）
- 跌停日：卖出信号 → 延迟到次日（或直接跳过）
- 在回测报告中标注"本回测未考虑涨跌停限制"

### 检查 5：停牌日成交

**风险**：停牌日成交量=0，但回测引擎是否仍然执行交易？

**当前实现**：
- `simulate()` 不检查 `volume[i] == 0`
- **结论**：❌ **未处理停牌日**。可能在不该有交易的日子成交。

**建议修复**：
- 在 `simulate()` 的每个 bar 循环中，`if volume[i] == 0.0 { continue; }`
- 在回测报告中标注停牌处理方式

### 检查 6：流动性限制

**风险**：大资金策略可能在成交量不足的股票上遇到流动性问题。

**当前实现**：不检查成交量与持仓规模的比例。
**结论**：⚠️ 当前仅支持小资金假设。可接受，但需在报告中声明。

## 三、交易成本检查

### 检查 7：手续费、印花税、滑点

**当前实现**（`lib.rs` BacktestConfig）：
```rust
commission_rate: f64,   // 默认 0.0003 (0.03%)
stamp_tax_rate: f64,    // 默认 0.001  (0.1%，A股单边卖出)
slippage: f64,          // 默认 0.001  (0.1%)
```

- ✅ 手续费双向收取
- ✅ 印花税仅卖出时收取
- ✅ 滑点对买卖均应用百分比
- ✅ 参数可配置

**建议**：在前端 UI 中加 tooltip 说明默认值的来源和含义。

## 四、数据处理边界

### 检查 8：数据不足

**风险**：短数据（<50 bar）是否会导致 panic 或无意义的回测结果？

**当前实现**：
- `simulate()` 中 `if signal_i < 0` 检查信号索引
- 部分策略在数据不足时返回空信号 → `total_trades = 0`，不会 panic
- **结论**：✅ 不会 panic，但短数据结果可能误导用户

**建议**：数据点 < 252（约1年）时，在回测结果区显示 "⚠️ 数据量较少（<1年），回测统计可能不稳健"

### 检查 9：缺失数据处理

**当前实现**：假设输入 DataFrame 中所有值的 `NaN` 已被预处理清除。
**结论**：⚠️ 上游数据清洗的责任不明确。

## 五、复权方式

**当前状态**：数据导入时使用东方财富 API 的 `kqt` 参数控制复权方式，默认前复权。
**结论**：✅ 复权方式在数据导入层处理，回测引擎不需要关注。
**建议**：在前端回测页面显示当前使用的复权方式（从数据元信息中读取）。

---

## 六、审计结论

| # | 问题 | 严重程度 | 建议 |
|---|------|:------:|------|
| 1 | 信号使用当日收盘价成交 | 🟡 中 | 在报告中标注假设 |
| 2 | 未处理涨停/跌停 | 🟡 中 | 标注限制 + 后续版本实现 |
| 3 | 未处理停牌日 | 🟡 中 | 标注限制 + 后续版本实现 |
| 4 | 短数据无警告 | 🟢 低 | 加数据量提示 |
| 5 | 无流动性检查 | 🟢 低 | 小资金假设可接受 |
| 6 | NaN 处理依赖上游 | 🟢 低 | 在 DataFrame 入口加校验 |

**总体评价**：回测引擎核心逻辑正确，无前向偏差。主要局限在于 A 股特殊规则（涨跌停/停牌）未处理，以及使用了"信号日收盘价成交"的假设。这些问题在当前阶段均可通过文档标注来解决，不阻塞 v0.12 发布。
