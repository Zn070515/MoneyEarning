## 三、关键指标（分两级：最低线 / 理想线）

| 指标 | 最低线（不阻塞发版） | 理想线 | 测量方式 |
|------|:---:|:---:|---------|
| 外部测试人数 | ≥ 1 人 | ≥ 5 人 | 测试记录表计数 |
| 测试设备数 | ≥ 3 台（含自己+虚拟机） | ≥ 5 台不同设备 | 设备矩阵记录 |
| 自测任务通过率 | 100%（8/8 自己在 3 台设备上通过） | — | 自己跑完 |
| 外部用户核心任务 | 至少完成安装+K线+理解定位 | 8 项全部通过 | 任务检查清单 |
| 首次 K 线可见时间 | ≤ 3 分钟 | ≤ 2 分钟 | 任务 2 计时 |
| 用户信任评分 | 无"这是荐股软件"反馈 | ≥ 3.5/5 | 直接询问 |
| 用户推荐意愿 | 无"绝对不会用"反馈 | NPS ≥ 0 | 直接询问 |
| 付费意愿（≥¥149） | 非零（至少 1 人愿意付费） | ≥ 30% | 直接询问 |
| SmartScreen 行为 | 有记录，知道跳过方法 | 100% 可跳过 | 设备矩阵 |
| 杀毒误报 | 有记录，知道哪些杀毒误报 | 0 次 | 设备矩阵 |
| Bug 阻断率 | 0 个 P0 阻断性 bug | 0 个 P0/P1 | bug backlog |
| 合规投诉 | 0 次 | 0 次 | 直接询问 |

> **找不到外部用户不是 v1.0 的阻断条件。** 真正阻断的是：(a) 用户误解为荐股软件 (b) 100% 不愿付费 (c) 不同设备上安装失败或闪退。

```
  ├─ 第一层：自己多设备自测（第 1-3 天，零外部依赖）
  ├─ 第二层：找 1-3 个熟人做 15 分钟核心体验（第 4-7 天）
  ├─ 第三层：网上自然分享获取反馈（第 8-14 天，长期策略）
  ├─ 发送安装包 + 简化测试任务
  ├─ 收集测试记录/反馈/信任评分/付费意愿
  └─ 填充 bug-friction-backlog.md

Week 5-6: 汇总分析 + RC + go/no-go
  ├─ 汇总所有数据到各文档
  ├─ 构建 RC，写 release-notes + known-issues
  ├─ 完成 sales-validation/（填入真实价格反馈）
  ├─ 完成 post-alpha/（v1.0 go/no-go 决策）
  └─ 提交 v1.0 go/no-go 结论
```

## 五、v1.0 go/no-go 决策公式

```
自测部分（权重 0.55，零外部依赖）：
  + 多设备自测通过率 × 0.25（≥3 台设备上 8/8 任务通过 = 满分）
  + 合规文案自查通过 × 0.15（逐页检查无荐股暗示）
  + 免责声明全部可见 × 0.10（回测/扫描/导出/启动 4 处）
  + 平台至少 1 个 GO × 0.05

外部验证部分（权重 0.45，有几人算几人）：
  + 外部测试通过率 × 0.15（完成的任务数/8，无人则用自测分数）
  + 无"荐股软件"误解 × 0.15（有外部用户反馈则用，无则用合规自查替代）
  + 付费意愿非零 × 0.10（有外部用户反馈则用，无则暂记为 0）
  + 无 P0 bug × 0.05

go 条件: 总分 ≥ 0.60 且 无合规投诉 且 至少 1 个平台 go
no-go 条件: 任一 pivot-or-stop 条件触发
```

> 外部用户数据缺失时，外部验证部分权重合并到自测部分（自测 0.70 + 平台 0.15 + bug 0.15 = 1.0）。这不是降级——独立开发者第一版就是这样的。

## 六、最终验收问答

完成后必须能逐条回答：

1. 自己在几台设备上跑通了全部 8 项任务？找到几个外部用户？他们的反馈是什么？
2. 用户（或自己隔夜自测）遇到的最大阻碍是什么？
3. 有没有人误解成荐股软件？（如果没找到外部用户：隔夜自测+假装你是你爸，能理解这是分析工具而不是荐股软件吗？）
4. 外部用户愿意付费吗？愿意付多少？（如果没找到外部用户：你自己觉得 ¥149/¥199 这个价格合理吗？为什么？）
5. 哪些平台可以发？哪些绝对不能发？为什么？
6. 有没有合规投诉？（如果没人在用：所有文案通过压力测试了吗？有没有红线违规？）
7. v1.0 是 go 还是 no-go？依据是什么？
8. 如果 go，前 10 个咨询怎么接？前 3 个订单怎么处理？
9. 如果 no-go，暂停还是转向？转向什么方向？
10. RC 安装包在 ≥ 3 台设备上验证通过了吗？
11. **如果没有外部用户：你确认产品没有致命问题（闪退/荐股误解/全平台封禁）吗？敢不敢自己先开始卖，边卖边改进？**
